# pi-git

A pi-coding-agent extension that provides interactive diff review and AI-assisted commit decomposition for git workflows.

## Diff & Review

**Hunk**:
A logical commit group consisting of related file changes and a single Conventional Commit message, generated by AI analysis of the working tree diff.
_Avoid_: chunk, patch, commit unit

**未割り当てファイル (Unassigned File)**:
A changed file that is not yet included in any hunk; it remains in the working tree for re-evaluation in the next review cycle.
_Avoid_: orphan file, leftover change

## Commit Workflow

**逐次承認 (Sequential Approval)**:
A review flow where hunks are examined, adjusted, and approved one at a time; each approved hunk is committed immediately before moving to the next one.
_Avoid_: batch review, bulk commit

## Flagged ambiguities

**Hunk**: In standard git terminology, "hunk" refers to a single `@@` block within a diff. In pi-git, it refers to a logical commit group spanning multiple files. When discussing git internals, use "diff hunk" or "@@ block" to disambiguate.

## Example dialogue

> **Dev**: I'm working on the new login feature and I also fixed a typo in the README. Should I commit them together?
>
> **Domain Expert**: Run `/git-diff`. The analyzer will probably split those into two hunks — one for the login feature and one for the docs fix. You can review each hunk in the file tree, move files between hunks if needed, and commit them separately through 逐次承認.
>
> **Dev**: What if the analyzer groups something I don't want together?
>
> **Domain Expert**: Select the file in the tree and toggle it out of the current hunk. It becomes an 未割り当てファイル and will be re-evaluated for the next hunk. After the current hunk is committed, the remaining changes are re-analyzed.
>
> **Dev**: And if I approve a hunk but want to tweak the commit message?
>
> **Domain Expert**: You can edit the message before committing. The AI-generated message is just a starting point.
